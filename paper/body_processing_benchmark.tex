\section{Data Processing Pipeline}

WhiteTiger transforms heterogeneous human-teleoperated robot operation records into a consistent dataset for large-scale policy training. Fig.~\ref{fig:data-processing-pipeline} summarizes the conversion from platform-specific HDF5 trajectories to model-ready LeRobot v2.1 records through validation, filtering, metadata standardization, embodiment-schema alignment, and format conversion. The pipeline preserves embodiment-specific control semantics, rejects invalid or task-inconsistent trajectories, and exposes a unified interface for robot learning systems.

\subsection{HDF5 ingestion and trajectory validation}

Raw WhiteTiger trajectories are organized as HDF5 records. Each record is associated with a robot platform, task identity, episode identity, and available modality fields. Source trajectories may contain visual observations, proprioceptive states, actions, task annotations, timestamps, and episode-level metadata, depending on platform and task setup.

Validation is required before trajectories enter the released training corpus. The pipeline checks temporal ordering, usable modality fields, task metadata consistency, platform metadata validity, and observation-action alignment. This stage removes incomplete, corrupted, or unusable trajectories that would otherwise create incorrect supervision during model training.

\subsection{Quality control and metadata standardization}

WhiteTiger uses a multi-stage quality-control workflow covering acquisition, monitoring, upload, validation, cleaning, review, and final delivery. At billion-frame scale, a small invalid fraction can correspond to a large amount of corrupted supervision. Quality control therefore constrains noise introduced by heterogeneous platforms while retaining the embodiment and interaction diversity required for foundation-model adaptation.

A trajectory is retained only when it satisfies the requester-defined task specification. Removal cases include incorrect scene setup, incorrect objects or tools, wrong manipulation procedure, incomplete execution, corrupted recording, missing modality fields, inconsistent task metadata, and observation-action misalignment. Failed or task-inconsistent demonstrations are not intentionally preserved as training examples in the released corpus.

Task and episode metadata are standardized for dataset analysis and model conditioning. Each episode is associated with task information, platform metadata, and source-specific fields when available. Manual task annotations provided at collection-request time supply language or semantic conditioning signals for vision-language-action policies. Metadata standardization checks that the recorded trajectory remains consistent with the intended task description.

\subsection{State and action schema alignment}

State and action schemas differ across robot platforms. Joint layouts, gripper structures, dexterous-hand dimensions, mobile-base commands, control frequencies, and action semantics are platform-dependent. WhiteTiger preserves schemas that identify arm joints, grippers, hands, bases, and other controllable components within each platform.

Schema alignment enables unified dataset loading without erasing native control spaces. It also defines evaluation metrics: Joint MSE uses only joint-related action dimensions, while ALL MSE uses the complete action vector for each platform. End-effector values are stored according to the physical travel range of each platform. WhiteTiger does not impose a universal gripper or hand scale across robots, preserving the action values used by the original controllers.

\subsection{HDF5-to-LeRobot v2.1 conversion}

After validation and schema alignment, source HDF5 records are converted into LeRobot v2.1. The conversion standardizes episode and frame organization, synchronized visual observations, robot states, actions, human-provided task annotations, event or keyframe annotations when available, and episode-level metadata. HDF5 remains the source trajectory container, while LeRobot v2.1 provides the training interface for dataset loading, episode management, and downstream evaluation.

WhiteTiger does not force every platform into an identical observation or action vector. Each robot retains its modality-specific state, action, and camera schema within the shared LeRobot v2.1 interface. This design supports consistent data access across platforms while preserving the physical control semantics required for cross-embodiment policy learning.

\section{Benchmark Experiments}

The benchmark evaluates whether WhiteTiger fine-tuning improves offline action prediction on external robot demonstration datasets. The evaluation datasets are excluded from the WhiteTiger training corpus. The protocol therefore measures dataset-level transfer from WhiteTiger fine-tuning to separately collected held-out trajectories, not reconstruction of WhiteTiger training data.

\subsection{Offline open-loop zero-shot evaluation}

Open-loop evaluation compares model-predicted actions with reference actions from human-teleoperated demonstrations under recorded observations. The benchmark compares a no-training GR00T N1.6 baseline~\cite{nvidia2025grootn1} with the same model after one epoch of full fine-tuning on WhiteTiger. The evaluation covers \textbf{13 platforms} and \textbf{42 paired task-dataset records} from datasets external to WhiteTiger and excluded from WhiteTiger training. No additional fine-tuning is performed on these evaluation datasets. Zero-shot therefore denotes dataset-level transfer from WhiteTiger training to external evaluation records.

\subsection{Evaluation protocol}

For each task-dataset record, five trajectories are randomly sampled from the corresponding external evaluation dataset. The same sampled trajectories are used for both models, producing a paired comparison between the no-training baseline and the WhiteTiger-trained model. The model predicts actions from recorded observations, and prediction error is computed against the associated human-teleoperation actions. Predictions use an action horizon of 30. Each trajectory is evaluated over its available length up to 3000 steps. The task-level result is the mean of trajectory-level MSE values over the five sampled trajectories.

Per-platform results are unweighted means over task-dataset records associated with each platform. The overall benchmark result is the unweighted mean across all 42 paired task-dataset records. This design gives equal weight to each task-dataset record independent of platform-specific data volume.

\subsection{Evaluation metrics}

All reported MSE values are normalized by action range to improve comparability across platforms with different action scales. For a trajectory with $T$ evaluated timesteps and a selected set of $D$ action dimensions, normalized MSE is defined as

\begin{equation}
\mathrm{NMSE}
=
\frac{1}{TD}
\sum_{t=1}^{T}
\sum_{d=1}^{D}
\left(
\frac{\hat{a}_{t,d}-a_{t,d}}
{a_{d}^{\max}-a_{d}^{\min}}
\right)^{2},
\label{eq:normalized-mse}
\end{equation}

where $a_{t,d}$ and $\hat{a}_{t,d}$ denote the reference and predicted action values for dimension $d$ at timestep $t$, and $a_{d}^{\max}-a_{d}^{\min}$ denotes the corresponding action range. The no-training baseline and the WhiteTiger-trained model use identical action normalization statistics.

\textbf{Normalized ALL MSE} is the primary evaluation metric. In Eq.~\ref{eq:normalized-mse}, $D$ includes the complete action vector, consisting of robot joint actions and end-effector actuation dimensions such as grippers or dexterous hands.

\textbf{Normalized Joint MSE} is an auxiliary diagnostic metric. In this case, $D$ includes only robot joint action dimensions and excludes end-effector actuation dimensions. The separation allows joint-motion prediction to be analyzed independently from platform-specific gripper or dexterous-hand control.

\subsection{Overall results}

One epoch of full-parameter fine-tuning on WhiteTiger reduces normalized offline open-loop prediction error across the 13 evaluated platforms and 42 external paired task-dataset records. At the aggregate level, the WhiteTiger-trained model reduces Normalized ALL MSE from 0.157283 to 0.005051, corresponding to a \textbf{96.79\%} relative reduction. It reduces Normalized Joint MSE from 0.116746 to 0.000680, corresponding to a \textbf{99.42\%} relative reduction. Fig.~\ref{fig:per-platform-zero-shot} reports per-platform comparisons.

These results show that WhiteTiger supplies supervision that improves full-vector and joint-related action prediction on evaluation trajectories excluded from the training corpus.

\subsection{Per-platform results}

Aggregate metrics are insufficient for a multi-embodiment dataset because action spaces, control frequencies, end-effector structures, and task distributions differ by platform. Per-platform analysis therefore reports Normalized ALL MSE and Normalized Joint MSE for each evaluated robot. Across all 13 platforms, the WhiteTiger-trained model obtains lower error than the no-training baseline under both metrics.

These results support the use of WhiteTiger as a shared adaptation corpus for heterogeneous robot embodiments.

\subsection{Task-level analysis}

Task-level results measure whether WhiteTiger fine-tuning improves normalized offline prediction across the 42 external paired task-dataset records. This granularity is necessary because platform-specific action spaces, control rates, and end-effectors can obscure task-level behavior in aggregate averages.

The largest reductions occur on records where the no-training checkpoint has high action prediction error. Residual errors after WhiteTiger fine-tuning remain small in absolute value but vary across platforms and tasks due to action representation, task dynamics, and end-effector control differences.

\subsection{Planned real-world evaluation of event-guided value learning}

The offline zero-shot benchmark evaluates the dense-trajectory function of WhiteTiger for foundation-model adaptation and cross-dataset action prediction. Event- and keyframe-level annotations target a separate question: whether explicit task-progress supervision improves closed-loop execution. A reserved real-world protocol will compare policies trained with and without event-derived value supervision.

Both policies will start from the same supervised fine-tuned model and will be evaluated on the same real-robot tasks under randomized initial configurations. The baseline policy will use dense trajectory supervision without event-derived value learning. The event-guided policy will use WhiteTiger event and keyframe annotations to construct task-progress supervision from subgoal and event completion. This design isolates the contribution of temporal annotations under real execution.

The planned evaluation will cover one to three robot embodiments and one to three representative manipulation tasks. Execution success rate will be the primary metric. Event completion rate, failure stage distribution, and time to completion will provide diagnostic measures. Until these experiments are completed, event-guided value learning remains a reserved evaluation direction rather than a demonstrated performance claim.

\subsection{Additional benchmark extensions}

The current benchmark provides dataset-level held-out offline zero-shot comparison. Natural extensions include data-scaling curves for policy scaling-law analysis, held-out-platform evaluation for cross-embodiment generalization, closed-loop simulation evaluation, and real-world rollout success-rate experiments for event-guided learning.
